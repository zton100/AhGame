# ADR-001: 项目基础技术选型

## Status

Accepted

## Date

2026-06-23

## Context

《深渊遗装》v1.0 是 Flutter/Dart 单机暗黑挂机刷宝手游。文档明确要求：

- 正式项目结构，不做临时 Demo。
- 所有玩法内容数据驱动，来自本地 JSON 配置。
- Flutter UI 与战斗、掉落、属性、离线收益等业务逻辑分离。
- 本地存档优先，后续预留云存档。
- 每个核心系统必须可测试。
- 先用小内容量跑通闭环，再扩完整内容。

当前需要确定首批实现的基础选型，避免后续每个系统重复讨论。

## Decision

采用以下默认方案：

1. 状态管理使用 Riverpod。
2. 本地存档首版使用 Hive。
3. 首批示例数据使用全部 5 个职业。
4. UI 首版目标是高质量可用骨架，复杂动效和正式美术后置。
5. 商业化边界严格按文档执行：只卖便利性、外观、仓库容量、BD 预设栏、离线上限，不卖毕业装备、核心词缀、直接通关。
6. issue 管理首版继续使用本地 Markdown 草案，待工程底座稳定后再决定是否迁移到 GitHub Issues。
7. 每个 Milestone 可作为父级工作包，具体开发按 `docs/开发需求拆解.md` 的 issue 粒度推进。
8. GitHub 远端仓库使用 `https://github.com/zton100/AhGame.git`，后续稳定增量应提交并推送到该仓库。

## Alternatives Considered

### Provider / ChangeNotifier

- Pros: 简单，Flutter 生态常见，入门成本低。
- Cons: 随系统增多后依赖关系容易变隐式，测试和组合复杂度上升。
- Rejected: 本项目系统多、状态来源多，Riverpod 更适合维护清晰的依赖边界和可测试 provider。

### SQLite

- Pros: 查询能力强，适合关系型数据和复杂检索。
- Cons: 对首版单机存档偏重；装备实例、设置、进度等对象型存档需要更多映射层。
- Rejected: v1.0 首要目标是稳定本地存档与迁移，Hive 更轻、更快落地。未来若需要复杂查询，可在仓库/服务层后面替换或补充 SQLite。

### 首批只做 1-2 个职业

- Pros: 最小实现更快。
- Cons: 无法完整验证职业限制、职业标签、职业装备权重、技能池分布、BD 覆盖和未来 25 套核心 BD 的结构。
- Rejected: 本项目的核心是多职业构筑。首批直接建立 5 个职业的轻量配置，可以更早发现职业模型、配置校验和 UI 展示的边界问题。内容深度仍保持最小，只为每个职业提供基础属性、标签和少量示例技能/装备。

### 首批直接追求完整视觉表现

- Pros: 更接近最终卖相。
- Cons: 在核心玩法闭环未稳定前容易返工。
- Rejected: 先做高质量可用骨架，保证布局、状态、弹窗、列表、品质表现稳定；复杂动效和素材替换后置。

### 立即迁移到 GitHub Issues

- Pros: 更标准的协作方式。
- Cons: 当前仓库尚未确认远程和 issue tracker 配置，过早发布会增加管理成本。
- Rejected for now: 先用本地 Markdown 作为可审阅、可复制的 issue 源，后续需要时再发布。

## Consequences

- 后续实现 Issue 001 时应直接加入 Riverpod 依赖。
- S03 存档实现默认围绕 Hive 设计，但 service 接口要隔离具体存储，以便未来迁移。
- `assets/data` 首批内容包含全部 5 个职业：`exile`、`necrospeaker`、`ember_mage`、`frost_ranger`、`sanctifier`。每个职业先提供轻量示例内容，用于验证职业限制、标签系统、技能池和装备权重。
- UI 需求中的 HITL 项默认按本 ADR 决策推进，只有品牌视觉、商业化策略发生重大变化时再重新确认。
- 商店系统不得引入破坏刷宝公平性的商品类型。
- 本地 git remote `origin` 指向 `https://github.com/zton100/AhGame.git`。远端当前可作为首批工程底座推送目标。
